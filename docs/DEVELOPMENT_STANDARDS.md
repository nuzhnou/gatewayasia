# Стандарты Разработки и Архитектурные Правила (Development Standards)

> **Автор**: Агент-Архитектор Систем  
> **Статус**: Утверждено  
> **Назначение**: Гарантировать гибкость, масштабируемость и легкое тестирование новых гипотез без разрушения существующей кодовой базы.

Любой разработчик или ИИ-агент, продолжающий работу над проектом, обязан строго следовать этим правилам.

---

## 🏗️ 1. Архитектурные принципы (Software Architect)

Мы следуем принципу **слабой связанности (Decoupling)** и **обратимости архитектурных решений (Reversibility)**. Код разделен на изолированные доменные слои:

```
┌────────────────────────────────────────────────────────┐
│                        INDEX.JS                        │
│                (Управление и планировщик)              │
└───────────────────────────┬────────────────────────────┘
                            ▼
┌────────────────────────────────────────────────────────┐
│                        SRC/BOT/                        │
│        (Интерфейс мессенджера — Telegram/Viber/etc)     │
└──────┬────────────────────┬────────────────────┬───────┘
       │                    │                    │
       ▼                    ▼                    ▼
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   SRC/AI/    │     │ SRC/DATABASE/│     │SRC/SCRAPERS/ │
│ (ИИ-Промпты) │     │ (SQLite БД)  │     │  (Парсеры)   │
└──────────────┘     └──────────────┘     └──────────────┘
```

### Правила разделения слоев:
1. **Изоляция ИИ-логики (`src/ai/`)**:
   - Никакого жесткого кодирования Telegram API или SQLite-запросов внутри файлов ИИ-агентов.
   - ИИ-агент должен получать на вход только историю диалога и новое сообщение, а возвращать чистый стандартизированный JSON-объект.
2. **Абстракция мессенджера (`src/bot/`)**:
   - Бот отвечает только за отправку/прием сообщений и обработку Telegram-команд.
   - Вся логика квалификации соискателя делегируется ИИ-агенту.
3. **Независимость Базы Данных (`src/database/`)**:
   - Все запросы к БД изолированы в модуле `db.js`.
   - Если мы решим заменить SQLite на PostgreSQL, Supabase или MongoDB, изменения должны затронуть **только** файл `db.js`.
4. **Обезличивание вакансий (Anti-Bypass Protection)**:
   - В вакансиях, передаваемых ИИ-агенту и публикуемых скрапером, запрещено указывать оригинальные бренды (Zara, LG, Amazon) и контакты Центра Легализации.
   - Названия заменяются общими терминами, а вакансия идентифицируется по ID (например, `PL-ZR-01`).

## 🔄 2. Моделирование процессов и сбоев (Workflow Architect)

Каждая интеграция и шаг в системе должны обрабатываться с учетом потенциальных ошибок и тайм-аутов.

### Контракт Квалификации Лида:
ИИ-агент обязан возвращать JSON-ответ по строгому контракту:
```json
{
  "reply": "Текст сообщения для кандидата",
  "qualification": {
    "candidateName": "Имя или null",
    "citizenship": "Гражданство или null",
    "documents": "Виза/биометрия или null",
    "targetVacancyId": "ID вакансии или null",
    "startDate": "Дата или null",
    "phone": "Телефон или null",
    "isFullyQualified": true/false
  }
}
```

### Обработка ошибок (Fault Tolerance):
1. **Сбои API-ключей Gemini**: При ошибке запроса к ИИ система обязана переключиться на безопасный fallback (предопределенные текстовые ответы кандидату с заглушкой) и записать ошибку в таблицу `logs`.
2. **Блокировки аккаунтов (Facebook/Telegram)**: 
   - Запрещено выполнять частые однотипные вызовы API.
   - Все публикации в соцсети должны происходить с псевдорандомной задержкой (от 10 до 25 минут) через резидентные прокси.
   - Каждый текст поста должен переписываться ИИ индивидуально (Text Spinning).
3. **Таймауты сетевых запросов**: Лимит ожидания ответа от Apify или Gemini API составляет 30 секунд. По истечении таймаута операция прерывается с записью в лог и отправкой уведомления админу.

## 🧪 3. Принцип непрерывного тестирования (TDD / API Testing)

Каждое архитектурное изменение или внедрение новой гипотезы должно сопровождаться проверочным скриптом.

1. **Запрещено коммитить/деплоить код**, ломающий тесты `test_phase1.js`, `test_phase2.js` и `test_phase3.js`.
2. **Создание новых тестов**: При добавлении новой функциональности (например, отправка анкет по SMTP) разработчик обязан создать `test_phase4.js` для её изолированного тестирования.
3. **Сухая проверка (Dry-Run)**: Все тесты должны поддерживать режим работы без валидных API-ключей (Mock-режим) для проверки синтаксиса и внутренней логики кода.

---

## 💡 4. Инструкция по тестированию новых гипотез

Если вы хотите внедрить новую гипотезу (из списка `docs/HYPOTHESES.md`):

1. **Создайте отдельный файл** в соответствующей директории (например, `src/marketing/smtp_sender.js` для отправки писем).
2. **Инициализируйте зависимости** и пропишите параметры в `config.json`.
3. **Создайте файл теста** `test_phase4.js` в корневой папке.
4. **Запустите изолированный тест**: `node test_phase4.js`.
5. **Интегрируйте готовое решение** в бота `src/bot/bot.js` или планировщик `index.js` только после того, как изолированный тест прошел успешно.
